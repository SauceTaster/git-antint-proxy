## commit-message-attribution

Channel scope is NON-tool-specific commit-message attribution; I deliberately excluded named-product trailers (e.g. specific AI assistant trailers) — those belong in the tool-specific catalog. Verified June 2026 against primary sources: GitHub docs (Co-authored-by format + noreply email forms), git-scm git-interpret-trailers and SubmittingPatches, kernel.org submitting-patches (Reviewed/Acked/Tested/Reported/Suggested-by + permission rules), developercertificate.org (DCO), Debian deb-changelog(5) manpage (trailer format with exactly two spaces before the RFC2822 date).\n\nVERIFIED FACTS: (1) GitHub noreply has two forms — modern ID+username@users.noreply.github.com (accounts after 2017-07-18) and legacy username@users.noreply.github.com; both link to a profile, so both are block-worthy. (2) Trailer keys are conventionally Capitalized-Dash-Delimited and colon-separated, parsed by `git interpret-trailers`; I allow optional spaces/dashes in keys to catch sloppy variants. (3) Debian changelog signature is '-- maintainer <email>  date' with one leading space and exactly two spaces before the date.\n\nSEVERITY RATIONALE: block = direct identity (name/email/handle-linkable trailers, DCO, email-quote lines, automated-gen markers); warn = traceable refs that need human review due to false-positive risk (bare emails, @mentions, ticket refs, changelog credits, robot emoji).\n\nFALSE-POSITIVE CAVEATS the proxy must handle: (a) email regex matches code/config samples and placeholder domains — I excluded example/test/invalid/localhost/domain TLDs and recommend redact-after-review. (b) @username collides with email local-parts, npm scopes (@scope/pkg), and Java/TS decorators (@Override) — negative lookbehind helps but warn-only is correct. (c) Jira-style ABC-123 collides with UTF-8/RFC-822/part numbers/SI units — strongly recommend scoping to known project prefixes or requiring an issue keyword (fixes/closes/refs) to fire. (d) Robot/sparkle emoji can be intentional Gitmoji, hence warn.\n\nREGEX NOTES: all detect values are JavaScript-flavored regex SOURCE strings (no enclosing slashes). Inline flag groups like (?im) are used; if the engine rejects inline flags, lift them to RegExp flags. Emoji are literal Unicode chars (🤖 U+1F916, 🧠 U+1F9E0, ✨ U+2728) in the source. The @username lookbehind requires lookbehind support (Node 9+/modern V8) — fine for a JS proxy. Multiline (m) needed for the anchored ^...$ trailer rules to match line-by-line within a multi-line commit message.

## identity-in-content

Scope: identity leakage inside committed file content and repo metadata (NOT the commit header/trailer channel). All formats verified June 2026 against primary docs: JSDoc/@author (jsdoc.app), Doxygen \\author, Google TODO(name) style, package.json person fields (npm docs), PEP 621 authors/maintainers inline tables (peps.python.org) plus legacy Poetry string array, Cargo.toml authors array (doc.rust-lang.org), git .mailmap format and AUTHORS/CONTRIBUTORS convention (git-scm), .npmrc init-author-* (npm docs), .gitconfig [user] (git-config), JetBrains .idea/workspace.xml and VS Code .vscode/ (vendor docs), and EXIF/IPTC/XMP image metadata (IPTC user guide, exiftool).

Detect strings are JavaScript-flavor regex sources (inline (?im)/(?i) flags used so each is self-contained) or globs. Backslashes are doubled for JSON; the proxy must JSON-decode before compiling.

False-positive guidance baked into each rule's fix. The highest-noise rules are content-email-address, content-phone-number, content-social-handle-mention, comment-by-name-attribution, and the absolute-home-path rules — all need allowlists (example.com/test/noreply domains; /Users/Shared, /home/runner, C:\\Users\\Public, runneradmin, vagrant; placeholder 'username'/'user' tokens; numeric collisions for phones).

Two-stage recommendation for assets: the binary-asset glob (asset-binary-with-metadata-glob) should SELECT files for an exiftool/metadata-strip pass rather than block, since the leak is in binary metadata not regex-able raw bytes; the asset-exif-author-metadata regex only catches text-extractable XMP/IPTC/EXIF strings. OOXML (docx/xlsx/pptx) and PDF carry their own creator/lastModifiedBy/Author fields handled by the same strip pass.

Severity rationale: 'warn' for high-signal identity (author tags, manifests, .mailmap, gitconfig, npmrc, home paths); 'info' for noisier or opt-in-shared cases (@handle mentions, generic author keys, .vscode shared settings, asset metadata). None set to 'block' because most are scrubbable in place rather than push-fatal; the orchestrator can promote per policy.

Adjacent secret note (out of strict channel scope but flagged in fixes): a committed .npmrc may also contain _authToken and .gitconfig may contain signing key references — worth a secrets pass alongside identity scrubbing.

## prompt-quote-llm-text

Scope: this channel catalogs verbatim-LLM-text leakage (assistant phrasings, prompt echoes, placeholders, chat residue) in committed code/comments/docs/commit messages - NOT structured attribution like commit trailers (that belongs to an agentic-trailer channel).

Severity philosophy: only unambiguous verbatim model boilerplate is "block" (self-disclosure "As an AI language model", knowledge-cutoff disclaimers, "Here is a commit message:", code-elision markers that would delete real code). Almost everything else is "warn" or "info" because the underlying words/phrases are legitimately used by humans. Single-word lexical tells (delve, robust, em-dash) are "info" and MUST be density/cluster-scored, never single-hit blocked, or false positives explode.

Key false-positive mitigations: (1) exclude *.md, *.mdx, *.rst, *.ipynb, and test/fixture paths for placeholder rules (foo/bar, example.com, your-api-key-here) and the markdown-fence rule - those tokens are normal there; (2) require co-occurrence of 2+ markers before hard-flagging high-risk lexical rules (\"Here's the\", \"It's important to note\", excess-vocabulary); (3) for the markdown-fence and elision rules, restrict to non-markdown source files.

Verified real-world grounding: refusal/disclaimer phrases confirmed leaking in actual repos - run-llama/llama_index #8842, lm-sys/FastChat #1211, google-gemini/gemini-cli #5477, and the OpenAI dev community thread. The escaped-prompt and elision markers (\"... existing code ...\", role labels, system-prompt phrasing) are documented in the leaked-system-prompts corpus (134k-star repo, per Augment Code writeup).

AI-fingerprint literature referenced (as requested): (1) arXiv:2508.16385 \"ChatGPT-generated texts show authorship traits that identify them as non-human\" - Dentella, Huang, Mansi, Grieve & Leivada (2025): LLM text shows limited register variation, noun-over-verb preference, distinct grammatical backbone, proposed as a litmus test for AI. (2) \"Your Large Language Models Are Leaving Fingerprints\" (ACL 2025 GenAIDetect workshop, aclanthology 2025.genaidetect-1.6). (3) Excess-vocabulary studies - arXiv:2406.07016 (biomedical) and PubMed PMC12679996 - quantify the post-ChatGPT spike in \"delve/underscore/intricate/meticulous\". (4) Originality.AI and ai-text-humanizer phrase lists for \"Certainly!/Absolutely!/As a large language model\".

Chat-template special tokens (<|im_start|>, <|im_end|>, <|endoftext|>, <|assistant|>) are folded into the transcript-role-label rule - these are near-zero-false-positive signals when found in committed files.

Implementation note for the proxy: these rules operate on diff content (added lines), commit subject/body, and full file blobs. Recommend running the lexical/info rules as a combined density scorer that emits one aggregate \"likely-LLM-text\" warn when N>=3 markers co-occur in a single comment block, doc paragraph, or commit message, rather than N separate low-value warnings. All regexes are JavaScript-flavor (note: JS does not support inline (?i)/(?im) flags natively - the proxy should translate the leading (?i)/(?im) into the RegExp 'i'/'im' flag arguments when compiling).

## temporal-behavioral

Grounding/verification (June 2026): (1) Horvath, Pietrikova, Spinellis, "Bridging Behavioral Biometrics and Source Code Stylometry: A Survey of Programmer Attribution," arXiv:2603.11150 (submitted 2026-03-11) — confirms the behavioral-biometrics framing (commit activity, edit timing, development sequences as identifying signals) but is a survey that explicitly states temporal/behavioral metrics are underexplored, so it frames rather than enumerates the offset/time-of-day rules; those specifics are grounded in the empirical sources below. (2) git-privacy by EMPRI-DEVOPS (https://github.com/EMPRI-DEVOPS/git-privacy) — VERIFIED exact git-config keys from README: privacy.limit (format 'hh-hh', e.g. 9-17, rounds a 17:30 commit to 17:00), privacy.mode (only 'reduce' supported, default reduce), privacy.pattern (chars: M=set month to January, d=set day to 1st, h=set hour to midnight, m=zero minutes, s=zero seconds), privacy.ignoreTimezone (bool; true=warn on tz change, false=abort), privacy.replacements (bool, default false). Hooks: post-commit (redacts new commits) and post-rewrite (handles rebase/redate). Also a redact-email subcommand. Requires Python >=3.9, Git >=2.22.0. The 'GitPrivacy:'-style recovery trailer key is configurable across versions - verify against the installed binary before treating the literal as authoritative. (3) Geographic Diversity in Public Code Contributions, arXiv:2203.15369, plus SWAN 2017 'Timezone and Time-of-Day Variance in GitHub Teams' (cmustrudel.github.io/papers/swan17.pdf) — UTC-offset-to-region inference validated at >90% of commits compatible with the contributor's declared country, and commit time-of-day rendered as circular histograms with 9am-5pm work-hours highlighted; these validate the tz-offset and time-of-day/weekday rules and confirm the schedule signal survives offset zeroing. (4) reproducible-builds.org (SOURCE_DATE_EPOCH spec and Archive metadata docs) — git archive stamps the commit time as each file's mtime; generated files embed compile-time timestamps; git does NOT reset working-tree source mtimes; SOURCE_DATE_EPOCH (UNIX seconds, UTC) is the canonical normalization, with GNU tar --mtime/--sort/--owner/--group flags for deterministic archives. (5) git-scm git-commit / git-commit-tree docs and the rebase date gist (gist.github.com/ugultopu) — commit-object identity line format is 'name <email> <unix-epoch> <+/-HHMM>'; rebase/amend/cherry-pick preserve author-date but reset committer-date to now, basis for the skew rule.

Detector scope: detect strings target the raw commit-object text (output of 'git cat-file commit'), where author/committer lines are 'name <email> <unix-epoch> <±HHMM>'. JS-regex compatible.

False-positive caveats: time-of-day, weekday/weekend, cadence, and sub-hour-precision rules are AGGREGATE/distribution signals - a single commit is not identifying, so per-commit regex FP risk is high; the detectors deliberately match broadly (any +0000 epoch) and the real enforcement is the quantization fix, not the match. future-date and archive-mtime detectors are coarse guards that must be backed by a runtime 'epoch > now' comparison and an unconditional --mtime override respectively, not the regex alone. The generated-file timestamp regex can match intentional documentation dates (medium FP); prefer toolchain SOURCE_DATE_EPOCH over blind redaction.

Recommended canonical normalization the proxy should apply to EVERY pushed commit: force both author and committer timezone offsets to +0000; set committer-date == author-date (zero skew); quantize the epoch (reduce seconds+minutes to a full hour minimum, optionally to a fixed working-hours bucket via a limit like 9-17, optionally to day granularity); clamp to [plausible-floor, now]; strip any git-privacy encrypted-date recovery trailer; and for build/release artifacts propagate a quantized SOURCE_DATE_EPOCH while forcing archive-member and source-file mtimes to it.

## stylometry-layout-lexical

SCOPE & GROUNDING. This catalog targets the LAYOUT and LEXICAL feature families of the Code Stylometry Feature Set (CSFS) defined in Caliskan-Islam et al., "De-anonymizing Programmers via Code Stylometry," USENIX Security 2015 (https://www.usenix.org/conference/usenixsecurity15/technical-sessions/presentation/caliskan-islam). The paper's Table 3 (Layout Features) and Table 2 (Lexical Features) give the EXACT feature names this channel is designed to flatten: ln(numTabs/length), ln(numSpaces/length), ln(numEmptyLines/length), whiteSpaceRatio, newLineBeforeOpenBrace, tabsLeadLines (layout); WordUnigramTF, ln(numKeyword/length), ln(numComments/length), ln(numLiterals/length), ln(numTokens/length), ln(numFunctions/length), avgLineLength, stdDevLineLength, avgParams, nestingDepth, branchingFactor (lexical). The 2026 survey by Horvath et al., "Bridging Behavioral Biometrics and Source Code Stylometry: A Survey of Programmer Attribution," arXiv:2603.11150 (https://arxiv.org/abs/2603.11150), confirms stylometric features still dominate the literature and that classical models (random forest, SVM) remain in heavy use alongside neural models, so flattening these surface features remains worthwhile.

CRITICAL HONESTY CAVEAT (must surface to the dev). Layout/lexical normalization is a PARTIAL defense only. Caliskan-Islam state on p.258 that the SYNTACTIC (AST) feature set is "invariant to changes in source-code layout, as well as comments," and on p.259 that "The AST node bigrams are the most discriminating features of all." Therefore running prettier/gofmt/black/rustfmt and normalizing whitespace/EOL/quotes does NOT defeat the strongest attribution signals: AST node bigrams (ASTNodeBigramsTF), AST node type frequency/TFIDF, code-unigram-in-AST-leaves, max AST depth, branching factor, and nesting depth (these are computed post-parse and survive reformatting). Several Table-2 "lexical" features also survive a formatter because a formatter does not rename identifiers, rewrite comments, or change token counts: WordUnigramTF (identifier/token n-grams), ln(numKeyword/length), ln(numComments/length), comment content, abbreviation habits, and average parameter count are untouched by prettier/black. Honest framing for the proxy: this channel raises the cost of layout/lexical-surface stylometry and removes formatter-defeatable signals, but a determined attribution model using AST features will still work. Defeating those requires semantic source-to-source transformation (identifier renaming, comment stripping, control-flow/AST rewriting), which is OUT OF SCOPE for a pure normalization pass and risks breaking code.

OPERATIONAL MODEL. The intended fix is canonicalize-to-configured-style on every staged blob before push: run the language's idempotent formatter (prettier for JS/TS/CSS/HTML/JSON/MD; black for Python; gofmt/goimports for Go; rustfmt for Rust) plus a deterministic whitespace/EOL/quote/import-order normalizer for languages/files with no formatter. Severity is "warn" across the board per task spec: the proxy flags "this blob would change under the canonical formatter" (i.e., it deviates from configured canonical style) rather than blocking. Most detectors below are pre-normalization heuristics on raw bytes/lines; the authoritative test is always "diff(blob, formatter(blob)) is non-empty," which has near-zero false positives but requires running the tool. The regexes are cheap pre-filters.

FALSE-POSITIVE NOTES. (1) Tabs/CRLF/trailing-whitespace detectors will fire on Makefiles, Go source (tabs are canonical), .gitattributes-forced-CRLF files, and patch/diff/markdown-hardbreak files — gate by file type and respect .editorconfig/.gitattributes. (2) Quote-style and semicolon regexes are noisy on raw bytes (strings, regex literals, template literals); the real arbiter is the formatter, so treat regex hits as advisory only. (3) Do not "normalize" inside string literals, heredocs, or vendored/generated/minified files — exclude node_modules, vendor/, dist/, *.min.*, and lockfiles. Config-file canonical-style sources to honor/emit: .prettierrc(.json/.yml/.yaml/.json5/.js/.ts)/prettier.config.js, pyproject.toml ([tool.black], [tool.isort] profile="black"), .rustfmt.toml/rustfmt.toml (tab_spaces, hard_tabs), and .editorconfig (indent_style, indent_size, end_of_line, insert_final_newline, trim_trailing_whitespace). gofmt has no config file and always emits tabs + LF; that IS the canonical Go style.
